## С чем я согласен

### 1. `I` сначала, `UP` и `B` отдельно
Да. Это правильная декомпозиция.

- `I` почти полностью механический и низкорисковый.
- `UP` уже может менять синтаксис и создавать спорные правки.
- `B` — это уже не только hygiene, а потенциально исправление реальных дефектов.

То есть `I` можно брать как hygiene PR, а `UP`/`B` лучше разбирать отдельно и осознанно.

### 2. CI важнее pre-commit
Да. Это его самый сильный аргумент.

Текущий репозиторий реально не запускает Ruff в CI: в `.github/workflows/wiki-lint.yml` есть `rebuild_index`, `lint --structural-only` и AST syntax check, но нет `ruff check` и нет `ruff format --check`.

Значит сейчас Ruff в проекте существует, но не является enforced gate. В таком состоянии pre-commit — только локальное удобство, а не защита `main`.

### 3. PEP 621 optional-dependencies лучше, чем uv-specific dev-dependencies
Да, с этим я тоже согласен.

Репозиторий действительно uv-first, поэтому это не принципиальный вопрос. Но если различие небольшое, стандартный путь лучше по умолчанию. `project.optional-dependencies.dev` даёт тот же результат без лишней привязки к конкретному tool-specific блоку.

### 4. `#2` (`[project.scripts]`) надо отложить
Да. Это по-прежнему самый невыгодный пункт по соотношению польза/риск.

В репозитории уже много жёстких ссылок на `uv run python scripts/wiki_cli.py ...`, так что это не маленькая cosmetic-правка, а совместимость-рефакторинг.

## Что Claude добавил правильно

### `.git-blame-ignore-revs`
Это хороший и важный довод. Я согласен.

Если делать первый массовый style-only commit после `ruff format`, то без `.git-blame-ignore-revs` blame действительно станет заметно хуже. Это не теоретическая проблема, а практическая деградация истории.

Официальные источники это подтверждают:

- Git docs: `git blame` поддерживает `--ignore-revs-file` и конфиг `blame.ignoreRevsFile`  
  https://git-scm.com/docs/git-blame
- GitHub Docs: файл `.git-blame-ignore-revs` в корне репозитория учитывается в web blame view  
  https://docs.github.com/en/repositories/working-with-files/using-files/viewing-and-understanding-files

Это значит, что его дополнение не просто «неплохая идея», а реально корректная часть первого массового format pass.

## Моя единственная оговорка

Я бы сформулировал это так:

- `.git-blame-ignore-revs` нужен **в тот момент, когда появляется первый большой style-only commit**
- не раньше и не как отдельная абстрактная housekeeping-задача

То есть логика правильная, но привязка должна быть к конкретному commit SHA, а не к общей подготовке заранее.
