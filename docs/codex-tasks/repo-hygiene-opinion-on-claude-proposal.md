# Мнение по предложению Claude о repo hygiene

Моё мнение: в целом предложение разумное, но приоритет и границы я бы немного поправил.

## 1. Finding #3 — dev-dependencies

Это действительно стоит делать первым. Риск почти нулевой, польза понятная: новые машины и новые сессии получают полный набор dev-инструментов без догадок.

Я бы выбрал **`[project.optional-dependencies]`**, а не uv-специфичный блок. Причина простая: это стандартный PEP 621 путь, он переносимее и не делает проект ещё более жёстко завязанным на `uv`.

## 2. Finding #1 — Ruff config

Это тоже разумная задача, но я бы не включал сразу пакет `I + UP + B`.

Сейчас в `pyproject.toml` у Ruff фактически только `line-length = 100`, то есть он почти ничего не проверяет. Усилить его стоит, но поэтапно:

- добавить `target-version = "py312"`
- сначала включить `extend-select = ["I"]`
- затем отдельно посмотреть `UP` и `B` по реальному `ruff check --diff`

Причина: `I` обычно почти безболезненный, а `UP` и особенно `B` могут уже дать заметный churn и спорные рекомендации.

## 3. Finding #4 — автоматический Ruff

Здесь я частично согласен. Автоматический запуск нужен, но важнее **CI/gate**, чем **pre-commit**.

Сейчас в `.github/workflows/wiki-lint.yml` Ruff вообще не запускается. Это означает, что главный enforced gate проекта про Ruff ничего не знает.

Поэтому я бы делал так:

1. сначала добавить `ruff check` и, возможно, `ruff format --check` в CI / gate
2. потом, если хочется локального удобства, добавить `.pre-commit-config.yaml`

Pre-commit полезен, но это локальный helper. CI — это реальный контракт репозитория.

## 4. Finding #2 — `[project.scripts]` для CLI

Это я бы отложил.

Теоретически идея нормальная: `uv run wiki <command>` короче и приятнее, чем `uv run python scripts/wiki_cli.py <command>`. Но в этом репо слишком много жёстких ссылок на текущую форму вызова — в документации, workflow и инструкциях.

То есть это уже не мелкое улучшение UX, а цепочка совместимых изменений. Польза небольшая, blast radius самый большой.

## Итог

Мой порядок был бы такой:

**#3 -> #1 (сначала только `I`) -> Ruff в CI/gate -> pre-commit при желании -> #2 сильно позже**

Коротко: Claude прав по направлению. Я не согласен только с тем, что pre-commit стоит ставить так же рано, и с тем, что `UP + B` нужно включать сразу одним заходом.
