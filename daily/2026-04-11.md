## [2026-04-11T01:15:06+03:00]

### Context
Testing "Pulse" messenger app on two physical Android devices (OnePlus + Samsung) via ADB, discovering critical registration/auth issues and ADB input limitations with Flutter. `project: messenger_test`

### Key Exchanges
- **Phantom session bug discovered**: Samsung was operating with phone number +79009876543 that **doesn't exist in the database** — previous test session registered incorrectly, creating a ghost session with no backing user record
- **Identity mismatch**: OnePlus was logged in as "Стёпа" (developer's real account), not "TestUser1" as assumed — no TestUser accounts existed in DB at all
- **ADB text input fails with Flutter TextField on Samsung**: typing via `adb shell input text` doesn't persist — field empties when focus changes. This is a known Flutter/ADB incompatibility
- **Workaround for auth**: bypassed Flutter UI entirely — set OTP manually in Redis (`"111111"`), then verified via direct API call to get auth token
- **Created test chat via API**: after auth workaround, created chat between "Стёпа" and "Дима" directly through REST API, sent test message successfully

### Decisions Made
- **Clear app data via ADB** (`adb shell pm clear`) instead of trying to navigate to logout through UI — faster and more reliable when UI interaction is flaky
- **Bypass Flutter UI for auth on Samsung**: ADB input is unreliable with Flutter TextFields, so authenticate via direct server API calls instead
- **Use existing real accounts** ("Стёпа" +79001234567, "Дима" +79003456789) instead of creating new test users — simpler path to functional testing
- **SMS_PROVIDER=mock** on server — OTP codes logged to server output, but easier to just set them directly in Redis

### Lessons Learned
- **Flutter TextFields don't work reliably with ADB `input text`**: the text disappears on focus loss. This blocks any ADB-automated UI testing of Flutter input fields on at least Samsung devices
- **Always verify DB state before testing**: phantom sessions (UI shows logged-in state but no DB record) can waste hours of debugging sync/contacts issues
- **Redis OTP retrieval is non-trivial**: key naming patterns may not match expectations; setting OTP directly via `SET` is more reliable than trying to read existing codes
- **Direct API testing is faster**: when device UI is unreliable, create chats/messages via REST API and only use devices to verify display/notifications
- **`/compact` is mandatory with ADB screenshots**: image-heavy sessions hit dimension limits quickly, blocking further interaction
