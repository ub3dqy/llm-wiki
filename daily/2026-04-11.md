## [2026-04-11T11:25:15+03:00]

### Context
Testing Pulse Messenger's messaging functionality (Stage 5-6) on two physical Android devices (OnePlus, Samsung) via ADB, project: messenger_test

### Key Exchanges
- **E2EE send failure investigated**: Samsung tried sending E2EE-encrypted message (transport_version=2 with envelopes) → server returned 400. OnePlus sent plaintext (E2EE not initialized) → 200 OK. Root cause: durable mutation path for encrypted messages fails server-side.
- **Message delete returns 500**: `DELETE /api/conversations/.../messages/{id}` → 500 Internal Server Error. `ExecuteDeleteMessageMutation` crashes. Outbox retries infinitely. Likely related to messages created via API curl having incompatible metadata for durable mutation delete.
- **device_id in JWT investigated**: Initially suspected missing `device_id` in JWT (line 1314: returns 400 "device context required" if `client_mutation_id` present but no `device_id`). JWT generation was confirmed correct — device_id always passed. Root cause was elsewhere (E2EE transport path).

### Decisions Made
- **Bug severity classification**: S5-2 (E2EE send 400) and S5-4 (delete 500) marked as **Critical**; S5-3 (plaintext fallback) as Medium; S5-1 (API messages no read receipts) as Low.
- **Continued testing despite critical bugs**: Rather than fixing S5-2/S5-4 immediately, chose to document and proceed with Stage 6 (groups/channels) to complete the test pass first.

### Lessons Learned
- **E2EE asymmetry between devices**: One device may initialize E2EE while another falls back to plaintext — both paths must be tested. The plaintext fallback silently works but defeats the purpose of E2EE.
- **Server logs are essential for diagnosing send failures**: Flutter logs weren't visible (release build), but server-side HTTP status codes + request paths revealed the exact failure point.
- **Durable mutations + E2EE interaction**: The durable mutation outbox path with `transport_version=2` (encrypted envelopes) triggers a different server code path that returns 400 — distinct from the plaintext send path that works fine.
- **API-created vs UI-created messages behave differently**: Messages created via curl API have different metadata than UI-created ones, causing delete mutation to crash (500). Delivery/read receipts also don't track for API messages.
- **Context limit hit during ADB testing**: After many screenshots, hit "exceeds dimension limit for many-image requests (2000px)" — session blocked. Confirms the `/compact` every 5-8 screenshots strategy from wiki.

### Action Items
- **Fix S5-2**: Investigate server handler for E2EE transport_version=2 messages via durable mutation — why 400?
- **Fix S5-4**: Debug `ExecuteDeleteMessageMutation` — likely metadata mismatch for certain message types
- **Investigate S5-3**: Why OnePlus E2EE service not initialized — is `_shouldUseE2ee` returning false?
- **Complete Stage 6**: Group/channel creation and messaging tests (interrupted by context limit)
